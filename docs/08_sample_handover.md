本文档用于样品交付、现场演示和最终验收前快速确认。当前阶段以稳定演示为目标，不再扩展新功能。

## 交付范围

| 部分 | 状态 | 说明 |
| --- | --- | --- |
| STM32 主控 | 可交付 | 本地刷卡、OLED、蜂鸣器、闸门控制、注册卡表 Flash 保存、ESP32 上报 |
| 座位端 STM32 | 可交付 | HX711 + 红外判断座位 1，PB8/PB7 判断座位 2/3，座位 LED 显示 |
| CC2530 ZigBee | 可交付观察 | 点对点座位状态上报，USART1 38400 |
| ESP32S3 网关 | 可交付 | WiFi、STM32 二进制协议、HTTP 主动上报 |
| 后端 | 可交付演示 | FastAPI 内存版状态和刷卡记录 |
| Web 前端 | 可交付演示 | 中文设备控制台，入口 `/library-seat/` |

## 启动顺序

1. 确认后端服务运行。
2. 确认 Web 前端可访问。
3. STM32 主控上电。
4. 座位端 STM32 上电。
5. ESP32S3 上电并连接 WiFi。
6. 浏览器打开：

   ```text
   https://www.kento.top/library-seat/
   ```

## 已知限制

1. 后端和 Web 当前用于演示，后端状态以内存保存为主。
2. STM32 主控只持久化注册卡表；通行记录断电清空，最终记录以后端/云端为准。
3. ZigBee 链路依赖 CC2530 模块地址、频道、PAN ID 和供电稳定性。
4. ESP32 上传做了节流和失败冷却，网络异常时不会影响本地刷卡，但 Web 数据可能延迟。
5. 当前不做登录、用户管理、预约系统和数据库持久化。

## 故障优先排查

1. Web 无 STM32 在线：先看 ESP32 串口是否收到 STM32 HEARTBEAT / CARD_EVENT，再看后端 `/api/iot/status`。
2. 刷卡本地正常但 Web 无记录：先看 ESP32 是否打印 `card-event POST OK`。
3. ZigBee 无座位数据：先用主控 `FF 84 00 FF` 看 `RX bytes / SEAT RX count`，再确认模块地址已对换为主控 `0x301E`、座位端 `0x0000`。
4. 主控疑似卡死：查看重启日志中的 reset reason 和 `FF E0 00 FF` 健康统计。
5. ESP32 不能访问云端：确认 `SERVER_BASE_URL_PROD` 为 `https://www.kento.top/library-seat`，并确认服务器 HTTPS 正常。
